# Decisions

## 2026-07-15 — Phase 6 UserPromptSubmit hook: shared hooks.json + in-hook harness gate (supersedes the ~/.codex manual install)
**Why:** The `session-start` hook proves the shared plugin `hooks/hooks.json` auto-wires on **both** harnesses (Codex reads it and sets `CLAUDE_PLUGIN_ROOT`) with zero manual step, and that a self-gating hook (emit only on a signal) is harmless where it shouldn't act. The earlier Phase 6 decision treated "an entry in the shared `hooks.json` fires in Claude" as blocking and routed the UserPromptSubmit hook through a manual `~/.codex/config.toml` install — which cost a manual step and, because the registration is a user-global absolute path, a stale-path footgun (point it at the wrong install and it runs a different-version hook). Superseded: wire the hook in the shared `hooks/hooks.json` under `UserPromptSubmit` (command `"${CLAUDE_PLUGIN_ROOT}/hooks/user-prompt-submit"`, so it always resolves to the *running* install's own copy — no drift), and keep it Codex-only-in-effect with a best-effort **in-hook harness gate**: silent under Claude (its UserPromptSubmit input carries `transcript_path`), fires under Codex (input carries `turn_id`), fires when the signal is ambiguous. The gate errs toward firing because the degradation is harmless — the hook is already silent whenever there's no active `.forge/runs/` run, exactly like `session-start`'s repo-signal gate; the only residual is that a recent Codex run could surface in a Claude session in that same repo if the discriminator ever fails. The `transcript_path`/`turn_id` discriminator is deferred live-verification on Codex (the field names could churn). Net: eliminates the manual install step and the stale-path footgun. Rejected: manual `~/.codex` install (a step + stale-path risk); firing in both with no gate (user requires Codex-only); harness detection via env vars (`CLAUDE_PLUGIN_ROOT` is set on both, not discriminating).
**Where:** hooks/hooks.json, hooks/user-prompt-submit, docs/forge/specs/2026-07-13-codex-exec-runner-design.md (§Session awareness + changelog), skills/planning/codex-execution.md, README.md

## 2026-07-14 — Phase 6 Codex runner session awareness: --notify + --status + Codex-only UserPromptSubmit hook (scope-widening split to Phase 7, stall watchdog dropped)
**Why:** A backgrounded `codex exec` runner was blind — terminal events (escalation, contract error, completion) and live state went unnoticed until a human checked. Phase 6 makes it observable three ways: `--notify` pushes a one-line summary on every terminal event (macOS `osascript` modal default on all three — completion included because it is the integration decision gate; `--notify CMD` override; `subprocess.Popen` fire-and-forget so a broken notifier never changes the exit code), `--status --run-dir` pulls a deterministic summary from `run.json`+receipts (dispatches nothing), and a Codex `UserPromptSubmit` hook auto-injects live state per prompt. **Designed net-new, not resurrected from `backup/4b-attempt`** (that plan combined session-awareness + scope-widening and predated Phase 5's commit discipline). Key scoping corrections made during design: (1) **Codex-only** — Claude Code already handles session awareness natively in-harness, so the hook is wired in the Codex surface (`~/.codex/config.toml [[hooks.UserPromptSubmit]]` or repo `.codex/hooks.json`, installed by documented copy) and **never** the shared plugin `hooks/hooks.json`, because both harnesses read that file (README: SessionStart works on Codex via the shared schema) so an entry there would fire in Claude — the exact cross-harness leak ruled out. (2) The **silent mid-run stalls that motivated the deferral were a Claude-dispatch failure, not the Codex runner** (which already bounds every subprocess with `--timeout`), so the stall watchdog is dropped here and belongs to the Phase 8 Claude-parity track. (3) **Scope widening** (`--widen`, verdict `widening` field, drafted `repair_task`, mechanical/decision classification) split out to Phase 7. Verified via web research that the Codex CLI has a real `UserPromptSubmit` hook with the identical `hookSpecificOutput`/`additionalContext` contract (Codex `notify` is side-effect-only, not injection) — so the hook is buildable; live firing is deferred verification on a Codex install (Codex CLI unavailable in the dev env; Phase 4 flag-check precedent), the hook logic fully unit-tested locally. Small core-loop change: `run.json` written incrementally (`running` at start → terminal status at end; `contract-error` marker when the run dir exists) so `--status`/the hook distinguish an in-progress run from a dead one and can report contract errors — chosen over receipts-only inference, which can't tell a live run from a killed one and can't see contract errors. `--status` + hook share `scripts/forge_status.py` (the hook imports it, never shells out). Rejected: adding UserPromptSubmit to the shared `hooks/hooks.json` (fires in Claude); a heartbeat watchdog (Claude-path concern, not this runner); resurrecting the combined 4b plan.
**Where:** docs/forge/specs/2026-07-13-codex-exec-runner-design.md (Runner, Receipts, Halt, new Session awareness section, Testing, changelog); implementation to follow in scripts/forge-run.py, scripts/forge_status.py, hooks/user-prompt-submit, skills/planning/codex-execution.md, README.md

## 2026-07-14 — Right-sizing forge's own source: forge-run.py + its test file decomposed by concern (behavior unchanged)
**Why:** Dispatched `forge-deep` workers stalled twice mid-Phase-5 in the ~80k-context band; a named contributor was forge's own oversized files — a worker editing runner logic ingested `scripts/forge-run.py` (1105 lines) + `tests/test_forge_run.py` (1542 lines) + the TDD skill + its brief (DEFERRALS 2026-07-14, part 2). Split both along their existing comment-delimited seams, zero behavior change, the full suite (118 pass / 1 skip) as the safety net held green throughout. `forge-run.py` (623 lines) keeps dispatch, review, the per-task rework loop, the plan loop, and the CLI; three concerns extracted to plain sibling modules it imports — `forge_plan.py` (plan parse/order/effort), `forge_git.py` (git helpers + review-packet assembly), `forge_receipts.py` (receipts/run.json/ledger) — over a `forge_common.py` foundation (the 5 dataclasses, tier/effort constants, and the single importlib load of the hyphenated `extract-brief`/`review-packet` re-exported as `eb`/`rp`). `forge_common` is imported as a *plain* module (`sys.path.insert(0, SCRIPTS_DIR)` at the top of forge-run.py) so `sys.modules` caches one instance — critical, because two importlib copies would give two `Task` classes and dataclass `__eq__` (which checks `__class__ is`) would silently fail test equality. forge-run.py re-exports every moved name into its namespace, so tests/docs keep addressing them as `forge_run.<name>`; the tests never monkeypatch a moved internal (they drive real temp git repos + a fake `codex` binary), so the split needs no seam changes. The 1542-line test monolith became a non-collected `tests/_forge_support.py` (fake codex, plan/spec fixtures, argv/verdict helpers, the importlib loader — exported via `__all__`, `from _forge_support import *`) plus six concern-focused `test_forge_*.py` files (commit, plan, dispatch, receipts, loop, review); pytest auto-discovers them, all 63 forge-run cases preserved. Rejected: leaving forge-run.py whole (task goal names source files, not only the test file); importlib-loading the new helpers (risks the duplicate-`Task` identity bug). Historical completed plans that reference `tests/test_forge_run.py` are left as-is (they record what was true then). This is harness-neutral cleanup, not a numbered roadmap phase.
**Where:** scripts/forge-run.py, scripts/forge_common.py, scripts/forge_plan.py, scripts/forge_git.py, scripts/forge_receipts.py, tests/_forge_support.py, tests/test_forge_{commit,plan,dispatch,receipts,loop,review}.py

## 2026-07-14 — Codex runner commits per passed task; clean tree required every invocation (phase 5)
**Why:** The runner never committed during a run, so the plan-level final review's `git diff HEAD` had no clean boundary — when the working tree started dirty (prior uncommitted phase work), every final review swept the entire accumulated cross-phase diff and judged unrelated code against this plan's spec, producing halt-grade findings the spec could never satisfy; re-runs compounded because fixes stayed uncommitted too. Root fix over a snapshot band-aid: the runner commits each passed task (`git add -A && git commit -m "forge: task N — <title>"`, empty stage skipped), so HEAD advances to a clean checkpoint after every task and `git diff base_commit` is exactly this plan's work. This restores the vertical-slice discipline the Codex runner had dropped and reaches parity with the Claude inline path (which already commits per task — emergent Claude Code behavior, not a forge convention, which is why the bug never surfaced there; the gap on Claude's *dispatch* path is tracked as roadmap Phase 8). Clean-tree precondition enforced at **every** invocation (first and resume), dirty → contract error exit 1: chosen over auto-reset-on-resume because a hard reset could silently destroy human fixes made between invocations — the runner never resets or stashes user work. Per-task review base becomes the prior commit (`_snapshot_worktree`/stash retired); final-review base is `base_commit`, persisted in `run.json` on first invocation and reused on resume so the diff spans the whole plan. Ledger annotation kept and rides in the task commit (git log is the parallel record). Env-gated skips (skip → exit 0 → counted green) addressed by a shared planning-skill authoring rule (assert infra present or make skip exit non-zero; skip ≠ pass) rather than fragile runner-side skip-marker detection — YAGNI, and harness-neutral so it also advances Phase 8. Rejected: snapshot-at-run-start (leaves the no-commit/no-slice/cumulative-diff problems intact), runtime skip detection, retiring the ledger annotation (bigger change than the bug needs).
**Where:** docs/forge/specs/2026-07-13-codex-exec-runner-design.md (Commit discipline + changelog), scripts/forge-run.py, skills/planning/SKILL.md (env-skip rule), skills/planning/codex-execution.md, README.md

## 2026-07-13 — Codex execution moves to a deterministic `codex exec` runner; machinery now justified; ultra effort prohibited
**Why:** Supersedes the execution half of the 2026-07-03 dual-harness decision. In-session dispatch testing surfaced the sharp edges directly: explicit-spawn friction, fragile model/effort routing, and completed child threads consuming quota — and the prose rework cap proved unenforceable (reviewers failed out, the process never stopped). The earlier "discipline over machinery" rationale rested on quota bugs being harness-side and unavoidable; `codex exec`'s process boundary eliminates them by construction, so the machinery objection no longer holds. `scripts/forge-run.py` (stdlib, sequential, whole-plan scope) owns dispatch, review, the rework cap as a loop counter, receipts, and ledger annotations; the conversational orchestrator only invokes, relays escalations, and holds gates — it cannot absorb work inline because it is not in the loop. Tier mapping updated to GPT-5.6 (trivial → luna/medium, standard → terra/high, complex → sol/high — Sol at high supersedes 5.5/xhigh on generation uplift); `ultra` effort prohibited at every tier because it spawns subagents inside the worker, breaking brief isolation; `max` is a human-only escalation, never a default. `codex/agents/*.toml` retired — `codex exec` takes model/effort as flags, and the worker contract text is sourced from `agents/*.md`, removing a manually-synced copy. Receipts stay ephemeral (`.forge/runs/`, gitignored) with plan checkboxes as the durable record, consistent with the 2026-07-02 ephemeral-artifacts rule. Rejected: per-task launcher and hybrid runner/conversation review splits — both re-expose the orchestrator to the drift the runner exists to remove; keeping in-session dispatch as a documented fallback — two paths to keep in sync, no environment that needs the second.
**Where:** docs/forge/specs/2026-07-13-codex-exec-runner-design.md, scripts/, skills/planning/codex-execution.md, codex/agents/ (deleted)

## 2026-07-11 — Plan-parsing scripts fail loud at brief generation; strict, single-form syntax over tolerant parsing
**Why:** Strengthens the 2026-07-02 "fail loudly over thin briefs" rule after two issues (#8, #9) showed the guarantee held only for *missing* input, not *malformed* input — the scripts silently absorbed structural drift, so a bad plan surfaced only when someone later noticed a review packet failed or a brief looked thin, the most expensive place to catch it. Standing constraint: `extract-brief.py`/`review-packet.py` accept exactly one documented form per field and raise at generation on any deviation, naming the real cause — never degrade, never guess intent. Concretely: task headings are exactly `### Task N:` (three #, colon), and `**Goal:**`/`**Spec:**` are each a single line — `**Spec:**` bare comma-separated heading names only (no parentheticals, no `;`, one `--spec` file). Rejected tolerant parsing (strip parentheticals, join wrapped lines) — it entrenches ambiguous syntax and re-introduces silent guesswork; the module's job is to eliminate model reading, so its failures must be legible to the author, not smoothed over. Fixes carry an authoring-side belt (the planning skill states each contract) plus the script-side enforcement — the script is the guarantee, the prose is prevention. Any future plan-parsing script inherits this: one form, loud failure, cause named.
**Where:** scripts/extract-brief.py, scripts/review-packet.py, skills/planning/SKILL.md, tests/test_extract_brief.py

## 2026-07-03 — Renamed theforge → forge; legacy signal accepted forever; migration is a hook nudge, not a script
**Why:** "theforge" was daily typing friction in every namespace (`theforge:planning`, update commands, `docs/theforge/`), and the agents were already `forge-*` — the "the" was a collision-avoidance artifact, not a choice. "forge" is generic in software (SourceForge, Laravel/Atlassian/Electron Forge) but genericness costs nothing at personal scope; alternatives (anvil, kiln, crucible) traded the intended meaning — the place where things get built — for distinctiveness nobody needs. The SessionStart hook accepts `docs/theforge/`/`.theforge/` indefinitely and appends a one-line `git mv docs/theforge docs/forge` nudge in legacy repos — rejected a migration script because the migration is one command (relative links inside the tree survive a parent rename). Historical plans/ideation docs keep the old name; the Phase 3 spec was amended (living-spec rule) since its manifest-name assertions became false.
**Where:** hooks/session-start, .claude-plugin/, .codex-plugin/, .agents/, skills/, agents/, codex/agents/, README.md

## 2026-07-03 — Codex dual-harness: same repo with dual manifests; divergence isolated to a reference file; discipline over machinery for orchestrator leaks and worker tracking
**Why:** Skills, scripts, and the session-start script are harness-portable verbatim, so a split repo or build step is machinery without benefit (superpowers v6 precedent). The Codex execution procedure lives in `codex-execution.md` loaded only when the Workflow tool is absent — the earlier "no reference files" rejection applies to rules that always bind; this one binds on one harness only. Codex lacks structural orchestrator/worker separation, so work-leak prevention is an explicit hard rule (orchestrator never edits implementation files; 2-iteration rework cap escalates to the user, never absorbs inline). Worker tracking: tier-prefixed `nickname_candidates` pools replace Codex's random names, and plan checkboxes double as the dispatch ledger — rejected any cleanup/lifecycle machinery because accumulation and quota bugs are harness-side (openai/codex #19197, #22779). Tier mapping mirrors intent, not price: gpt-5.4-mini/low, gpt-5.4/high, gpt-5.5/xhigh; agents install by documented copy because Codex plugins cannot bundle agents.
**Where:** docs/forge/specs/2026-07-03-phase3-codex-dual-harness-design.md, codex/agents/, skills/planning/codex-execution.md

## 2026-07-02 — Two stdlib Python scripts with ephemeral output; spec sections declared on the task at plan time
**Why:** Scripts must eliminate model reading, not typing: extract-brief.py assembles worker briefs (plan header + task block + declared spec sections), review-packet.py bundles task metadata + git diff for reviewers — plan, spec, and diff content never transit the orchestrator. Spec sections are declared per-task via a `**Spec:**` heading-list line written when the planner has full context, so extraction is fully mechanical (rejected: CLI-arg section refs at dispatch — requires the orchestrator to read the spec, the exact reading being eliminated). Python stdlib over bash/awk for parsing robustness; `--out` defaults ephemeral, nothing committed (rejected: committed briefs dir — per-dispatch artifacts polluting target-repo history). Both scripts fail loudly over emitting thin briefs.
**Where:** docs/forge/specs/2026-07-02-phase2-execution-efficiency-design.md, scripts/

## 2026-07-02 — Three-gear pipeline; gear 2 lives inside the brainstorming skill
**Why:** Proportionality at pipeline level: gear 1 = trigger floors (unchanged), gear 2 = delta to an already-spec'd system (conversational design, one gate, straight to tdd — no spec/plan files), gear 3 = full flow for new architecture. ## 2026-06-10 — TDD infrastructure gate: harness creation is plan-level work, never drive-by
**Why:** Third superpowers sink (after embedded code and 3-agent review): the near-verbatim tdd skill had no trigger floor, so a trivial edit to an untested artifact ran the full Iron Law — observed in Codex as 1.5 hours bootstrapping a test stack for a single 67kb HTML file. The gate keys on sanctioned scope, not harness existence: existing suite → use it, never a parallel stack; approved plan including test setup → build it (new codebases get tests this way, through the plan gate); ad-hoc edit with no harness → ask one question (set up testing vs. verify manually). Trivial mechanical edits don't trigger the skill at all, and the don't-trigger exclusions now live in the frontmatter descriptions — the always-loaded trigger surface — not just the skill bodies.
**Where:** skills/tdd/SKILL.md, skills/brainstorming/SKILL.md

## 2026-06-10 — Session-independent model/effort routing via per-task tiers and three depth-profile agents
**Why:** Subagents inherit the session's model *and* effort by default, so an accidental `/model` switch (or a deliberately cheap session) silently degrades plan execution. The planner tags each task trivial/standard/complex by characteristics — not category — at plan time (re-checked in self-review), and tags route absolutely: forge-light (haiku), forge-standard (sonnet/high), forge-deep (opus/xhigh). Agent frontmatter is the only mechanism that can pin effort, which is why the plugin ships agent definitions despite the lean ethos. Routing is disclosed in the execution offer; the user overrides at the gate. A dedicated router agent was considered and rejected — it has less context than the planner and adds a hop.
**Where:** skills/planning/SKILL.md, agents/
