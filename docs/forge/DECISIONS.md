## 2026-07-17 — Phase 12a (Claude mechanical parity): agent effort drift + stale reviewer prose + dispatch commit discipline
**Why:** Phase 12 splits into 12a (mechanical, this entry) and 12b (the Phase 7 disposition/convergence canon on the Claude dispatch path — its own spec). 12a lands the low-risk residue the Phase 8→10–12 decomposition left on the Claude path. (1) **Effort drift:** the 2026-07-16 tier recalibration dropped the effort defaults in SKILL.md's routing table and `forge-run.py`'s `TIER_MAP`, but agent frontmatter — the **only** mechanism that pins effort on the Claude subagent path — still carried the old strong settings, so Claude dispatch silently ran above the calibrated tier. Corrected: `forge-deep` `xhigh`→`high`, `forge-standard` `high`→`medium` (`forge-light` has no pin — haiku default). The removed strong settings remain the human's on-halt bump target, unchanged. (2) **Stale reviewer prose:** the recalibration retired the strength-escalation second reviewer (reviewer = task tier + fresh context, not a stronger model), but `forge-deep.md` still cast itself as "the escalation reviewer … after a first review finds substantive issues" **and** claimed the final-integration-reviewer role unconditionally. Both corrected: the escalation-reviewer role is deleted; the final-integration role is conditioned on "**when the plan's highest tier is complex**," and `forge-standard.md` now states it is the final-integration reviewer for an all-standard plan. This pins the reviewer-tier rule the skill already encodes (SKILL.md: final review = plan's highest task tier) so the agent files can't drift back to implying always-deep — a complex final reviewer is earned by a plan already carrying a complex task's named decision, not a default. (3) **Dispatch commit discipline (Half A):** the Claude **dispatch** branch never stated the clean-tree precondition + commit-per-task that inline already had and the Codex runner got in Phase 5 — so a dirty tree could sweep the whole cross-phase diff into every review (the over-scoping bug Phase 5 killed for Codex). SKILL.md's dispatch branch now requires a clean tree at run start (dirty → halt, never reset/stash user work) and commits each passed task, establishing the clean per-task review base (`git diff <prior commit>`) that 12b's runner-verified provenance will depend on — which is why 12a precedes 12b. On the Claude path the orchestrator commits after each task's review passes; on Codex the runner does. Rejected: folding 12a into 12b (mixes a mechanical cleanup with a design-heavy refactor, against the no-mixed-implementations constraint); leaving the final-integration role unconditioned in the agent files (re-introduces the always-deep drift the skill already fixed).
**Where:** agents/forge-deep.md, agents/forge-standard.md (frontmatter effort + reviewer prose), skills/planning/SKILL.md (dispatch commit discipline), docs/forge/ROADMAP.md (Phase 12 → 12a done / 12b planned). 12b (the Phase 7 canon port via a shared `scripts/` disposition helper the orchestrator drives) gets its own spec.

## 2026-07-17 — Inline execution self-reviews (no separate fresh-context reviewer); Phase 11 adds a human gate to inline, not an independent reviewer
**Why:** Fresh-context review earns its cost by removing implementer bias — an independent pass that doesn't rationalize the work's own defects (2026-07-16). That value scales with **design content**, but the dispatch cost is ~fixed, so the cost/benefit is worst exactly where inline operates — the **simple** end (inline is chosen because the change is simple and accumulated context is an asset). The system already trusts the simple end without a separate reviewer: the trivial tier skips subagent review and treats passing acceptance commands as verification enough (`SKILL.md`). The real objective, unbiased check on simple work is **TDD + acceptance commands** (tests don't rationalize), and that holds regardless of who reviews. So inline does **not** dispatch a separate reviewer: the session implements with context intact, TDD + acceptance carry the objective load, and the orchestrator self-reviews as a lightweight consistency pass. What inline gives up is the catch on *rationalized judgment calls* — small where inline lives, and backstopped by the Phase 11 human gate. **Consequence for the phasing:** Phase 11's inline fix is **not** "add a fresh reviewer" — it is "even in self-review, don't silently auto-resolve **decision-grade** findings; classify them (Phase 7 disposition canon) and surface the halt-grade ones to the human gate." Fresh-context review stays a **dispatch-only** concern (high bias, high stakes → worth the toll). The one live risk is inline creeping onto non-simple work where the missing bias-check bites; the guard is keeping the inline trigger honestly gated to simple/small, with the human gate as the net. Rejected: forcing fresh-context review on inline (pays a fixed toll to remove a bias that's minimal at the simple end, and throws away the accumulated context that was the reason to go inline); self-review with no gate (the current over-resolution flaw itself — the thing Phase 11 fixes).
**Where:** docs/forge/ROADMAP.md (Phase 11 line); to be specified in the Phase 10 spec (inline self-reviews) and the Phase 11 spec (gated self-review). Carries forward to Phase 11.

## 2026-07-17 — Phase 8 decomposed into Phases 10–12; build inline symmetry first, then fix the shared process, then finish Claude parity
**Why:** A parity audit — Codex as the reference implementation of the post-Phase-7 target state, bring Claude in line — surfaced two things the original Phase 8 charter missed. (1) The 2026-07-16 tier recalibration **half-landed on Claude**: SKILL.md's routing table and `forge-run.py`'s `TIER_MAP` dropped to the recommended effort defaults, but the **agent frontmatter — the only mechanism that pins effort on the Claude path — still carries the old strong settings** (`forge-deep` xhigh, `forge-standard` high), so Claude dispatch silently runs above the calibrated tier. (2) More consequentially, **Codex has no inline execution path**: the 2026-07-13 move to the runner (built to fix in-session *dispatch* — spawn friction, quota accumulation) made the runner Codex's *only* path, sweeping in the low end (simple edits, doc updates, mechanical changes, small plans) that never needed dispatch machinery. Since the inline/dispatch decision is one shared **task-shape** criterion (context-is-an-asset / simple change) — not a harness-capability decision — inline should exist symmetrically on both harnesses, carrying the same canonical process. Chosen sequencing (**build-flawed-first**, not build-it-perfect): Phase 10 adds the Codex inline *vehicle* mirroring Claude's **current (over-resolving)** inline model — structural path only, reaching symmetry; Phase 11 then fixes the finding-process **once in the shared home** (replace over-resolution with the Phase 7 disposition canon: fix/defer/halt + human gate), correcting both inline paths by construction; Phase 12 finishes the Claude-specific remainder (dispatch commit/clean-tree + the effort drift + stale reviewer prose). Rationale for the order: establishing symmetry against a *known* model is low-risk, and improving a *symmetric pair* once makes "fixes both" clean and provable — the inverse of entangling "add the path" with "redesign the process." Inline is **not** a stripped path: TDD and review-before-commit are canon regardless of mode; the only thing inline changes is the execution vehicle (session does the work vs. dispatch), and the only reason to pick it is that the change is simple. Rejected: forcing all of it into one Phase 8 (entangles the Codex-inline discovery with the Half-B redesign); building Codex inline correct-from-the-start (couples path-creation to process-redesign, and offers no symmetric pair to prove the fix against); leaving Codex runner-only (the low end has nowhere lightweight to go). **Where:** docs/forge/ROADMAP.md (Phase 8 → deferred/decomposed; Phases 10–12 added); implementation to follow in skills/planning/SKILL.md (mode-before-harness routing, shared inline canon), skills/planning/codex-execution.md (Codex inline branch), agents/*.md (Phase 12 drift fixes), and per-phase specs under docs/forge/specs/.

## 2026-07-16 — Phase 7 scope autonomy: a fix/defer/halt disposition matrix + convergence-based rework, replacing 4b's binary classify + count cap
**Why:** Two operator observations drove the reframe. (1) Every run that hit the rework cap of 2 was resolved by *one more manual run* — the carried findings were already in the brief, so the "escalation" was a counter tripping on **converging** work, not a considered human gate. (2) Final review had **no fix loop at all** ([forge-run.py:611](scripts/forge-run.py#L611), "findings are a human gate") and its findings were waved off as "out of scope of the phase" — which was really three different things collapsed into one halt. **Core insight: an autonomous fixer's failure mode is over-fixing, and over-fixing is diff over-scoping wearing a new hat — the exact bug Phase 5 killed.** So the design biases hard toward *not* fixing. Two independent axes classify every finding: **provenance** (in-diff vs pre-existing, **runner-verified against the actual diff line ranges, not trusted from the reviewer**) × **contract impact** (contract-breaking vs improvement, where contract-breaking **requires a named acceptance criterion** or the runner downgrades it to a deferral — the tier-policy "named evidence" rule). The quadrants: in-diff×contract-breaking → **fix** (the only auto-fix cell); in-diff×improvement → **defer strict** (no gold-plating our own new code — the anti-over-scoping guard); pre-existing×contract-breaking → **halt** (a real scope decision, carries a drafted repair task); pre-existing×improvement → **defer** (harmless, logged). This answers the operator's sharpest question — *harmless vs impactful deferral* is exactly the contract-impact axis; the impactful one (pre-existing but contract-breaking) is the one quadrant that must never be silently deferred **or** silently fixed, so it halts. **Spin guard:** the raw count cap is replaced by **convergence** — the reviewer labels each finding resolved/carried/new against the prior attempt (supplied in the re-review packet), and the runner decides deterministically: halt on a **regression** (a runner-tracked resolved id reappears, or acceptance goes green→red — the "shuffling one bad state to another" case) or **stuck** (a fix finding carried across two attempts); pass when no fix findings remain; otherwise rework. Net progress each round is not required (incremental reviewer discovery is allowed); a backstop of **5** (raised from 2) is only a seatbelt against slow non-convergence. Final review gains this **same loop**. Auto-fix is the **default** (`--autofix auto`, matrix applied) with `--gate` (halt on any finding) as the conservative escape hatch — the operator's read that the manual stop mostly isn't needed. A **terminal doc-sync stage** runs after all-green (operator: "something always feels missed") — reconciling *existing* docs to the shipped diff, never authoring new docs (that would be the gold-plating the matrix forbids). Rejected: **4b's binary mechanical/decision + auto-append repair task** — the binary couldn't express harmless-defer vs harmful-halt, and auto-appending tasks is scope creep the operator now explicitly wants gated; **a higher fixed cap** (2→4) — still can't tell converging from churning, just cuts the cord later; **pre-review doc-sync** (the ROADMAP's original framing) — the matrix already stops doc drift from halting (it's improvement→defer), and the operator wanted a *terminal* all-green reconciliation instead; **auto model/effort escalation on stall** — the operator chose plain progress-based rework; the effort bump stays a human on-halt action (the tier-policy floor is the *target*, not an automatic mechanism here). Codex-only (`forge-run.py`); the Claude path is Phase 8 (DEFERRALS 2026-07-13) — its **own** spec per the no-mixed-implementations constraint, not an amendment to the Codex-runner spec.
**Where:** docs/forge/specs/2026-07-16-phase7-scope-autonomy-design.md (new), docs/forge/ROADMAP.md (Phase 7 spec link); implementation to follow in scripts/forge-run.py (disposition matrix, convergence stop, final-review loop, `--autofix`, doc-sync stage, deferral aggregation, provenance verification), scripts/forge_common.py (`MAX_ATTEMPTS`→`MAX_ATTEMPTS_BACKSTOP=5`), scripts/review-packet.py (`--prior-findings`), tests/, skills/planning/codex-execution.md, and a changelog pointer in docs/forge/specs/2026-07-13-codex-exec-runner-design.md

## 2026-07-16 — Tier policy recalibration: standard is the floor, up-tier needs a named decision, effort defaults drop to provider-recommended
**Why:** Both harnesses over-provisioned at every tier — the complex tier ran the top model at near-top effort (Claude opus·**xhigh**; Codex sol·**high**, sol also the priciest at $5/$30). This is the reflex both providers explicitly warn against: Anthropic's Opus 4.8 guidance is "start at `high`, not `xhigh`"; OpenAI's Codex guidance is "default to terra·medium, start sol·medium, escalate only the stage that failed," and Sonnet 5 / Terra are near-Opus/near-Sol on coding at a fraction of the cost. Root cause: classification was a *prediction* ("how hard is this?"), and guessing-high felt safe, so tasks drifted up. **Reframe: tier is a floor + evidence, not a prediction.** (1) **Standard is the default**; the burden of proof is on *leaving* it — down to trivial requires demonstrated mechanicalness, up to complex requires a **named** design decision or cross-cutting invariant, persisted in the task's `Tier:` line and surfaced in the execution offer. File count, category, "touches core", "feels complex" are explicitly **not** evidence. (2) **Effort defaults drop to each provider's recommended start:** trivial luna·medium→low (haiku unchanged); standard sonnet·high→**medium** / terra·high→**medium**; complex opus·xhigh→**high** / sol·high→**medium**. Cross-harness asymmetry (opus·high vs sol·medium) is intentional — each is its own provider's stated default. The removed-as-default strong settings (xhigh/max, sol·high) are **not** deleted; they stay the target a human may bump to on a halt — unchanged. (3) **Reviewer = task tier + fresh context, not a stronger model** — the operator's insight: the reviewer's value is an independent pass that doesn't rationalize the work's own defects, so `REVIEW_MAP` is **retired** (collapses into `TIER_MAP`, resolving its stale-drift hazard), the strength-escalation second reviewer is removed, and the final integration review runs at the plan's **highest task tier** (not a pinned ceiling). A reviewer finding an issue is the loop working, **not** a failure — only exhausting the rework cap is, and that already halts for a human. Enforcement splits by capability: the runner checks justification **presence** (loud contract error on an off-floor tier with none); justification **quality** (real decision vs. category) is the planning-skill self-review's job, since a script can't judge it. Rejected: **absolute floor = trivial** (both up-tiers gated) — risks under-powering everyday tasks into rework churn (OpenAI's "cheaper turn costs more per accepted result"); **keep standard at high** — the default tier is the highest-volume, and provider guidance puts the everyday default at medium; **strong-reviewer premium** — strength was never the reviewer's value, fresh context is; **auto-escalation in this spec** — the mechanism that bumps a halted task belongs to Phase 7 (Codex) / Phase 8 (Claude). Cross-harness by nature, so it gets its **own** spec and does not ride the Codex-runner spec (respecting the logged "no mixed implementations" constraint) — it instead *supersedes* that spec's Tier-mapping + reviewer rows.
**Where:** docs/forge/specs/2026-07-16-tier-policy-recalibration-design.md (new), docs/forge/ROADMAP.md (new tier-policy entry + Phase 7 rework-cap note); implementation to follow in skills/planning/SKILL.md (Tier §, `Tier:` field, self-review directional burden, Execution routing table, offer disclosure, reviewer/final-review rules), skills/planning/codex-execution.md (reviewer routing, REVIEW_MAP retirement, final-review tier), scripts/forge-run.py (TIER_MAP recalibration, REVIEW_MAP removal, Tier justification presence-check, max-tier final review), tests/, and an amendment (changelog pointer) to docs/forge/specs/2026-07-13-codex-exec-runner-design.md

## 2026-07-15 — Codex runner gains a live monitor (Phase 9); reverses the monitor-terminal rejection
**Why:** The 2026-07-15 foreground-execution decision rejected the monitor terminal as YAGNI, on the belief that foreground execution met the visibility need. Live testing showed it doesn't: "foreground" is a blocking subprocess loop (`subprocess.run(capture_output=True)` per task) that swallows the worker's `codex exec` stream and emits two coarse stdout lines per task — a task can run minutes with no output, indistinguishable from a hang, and there's no view of *where* in the plan you are. Reversal is warranted because the premise (foreground = visibility) was falsified. Chosen approach: an **attach-from-outside observer** — the runner tees each subprocess's stream to `run_dir/task-N-live.log` and adds a live progress pointer (`current_task`/`current_phase`), run/task timestamps, `pid`, and a heartbeat to `run.json`; a new read-only `scripts/forge-monitor.py` renders a two-panel `rich` TUI (plan ledger + the in-flight task's live stream) run in a second terminal, with a full-width semantic banner on terminal state (red-orange halt / green complete). Rejected: the **wrapper model** (monitor launches the runner and owns its stdout) — bakes the monitor into the critical path and couples it to being the entry point; the observer keeps one source of truth (the run dir) and lets you attach/detach freely. Rejected: **Textual** — read-only monitor needs no interactivity; plain `rich` + `Live` suffices. The tee is near-free (the stream is output `codex exec` already produces and the runner currently discards) and is behavior-preserving for exit codes/tails/timeouts (the reviewer-crash stderr tail is read from the tee'd file, not lost — an explicit requirement). Also resolves DEFERRALS 2026-07-15 (hard-killed run stuck `running`): the heartbeat + `pid` let the monitor render a stale `running` run as `stalled?` rather than a perpetual live spinner. Visual identity ("Instrument" — `box.SQUARE` hairlines, cyan accent, dense ledger) locked with the operator via a rendered mock. Codex-only; the Claude path stays Phase 8. Live `codex exec` stream texture is deferred verification on a Codex box (dev env has no CLI; Phase 4 precedent).
**Where:** docs/forge/specs/2026-07-15-forge-run-monitor-design.md (new), docs/forge/ROADMAP.md (Phase 9); implementation to follow in scripts/forge-run.py (tee call sites), scripts/forge_git.py/forge_receipts.py (run.json fields), scripts/forge_status.py (read_run_state extension), scripts/forge-monitor.py (new), tests/, skills/planning/codex-execution.md, README.md

## 2026-07-15 — Phase 6 push machinery removed; session awareness is foreground execution, not notifications
**Why:** Live Codex testing showed the entire push half of Phase 6 was solving the wrong problem the wrong way. Two hard failures: (1) `osascript` modals never fired in Codex because the runner runs inside Codex's sandbox (Apple Seatbelt blocks Apple Events / GUI), and `fire_notify` is fire-and-forget to `/dev/null` so the silent denial was invisible; (2) the `UserPromptSubmit` hook errored `exit 127` on every Codex prompt (the shared-`hooks.json` `${CLAUDE_PLUGIN_ROOT}` command form didn't execute there). Stepping back with the operator clarified the actual need — *"when the runner halts and needs input, I want to see it in the session window when I check back"* — and that need is fully met by **running the runner in the foreground**: the orchestrator blocks on the command, so a non-zero exit (escalation 2 / contract error 1) returns control immediately and the orchestrator relays the halt into the conversation. The silent-halt failure mode was always a *backgrounding* artifact (fire `… &`, move on, nobody holds the exit code); foreground removes it by construction, no push needed. Removed: `--notify`/`fire_notify` (+ per-task `task-passed`, the four-event set, `FORGE_NOTIFY_DISABLE`, `tests/conftest.py`), the `hooks/user-prompt-submit` hook (+ its `hooks.json` wiring, `test_prompt_hook.py`), and the now-dead `forge_status.render_hook_block`/`_compress_tasks`. Kept: `--status --run-dir` + incremental `run.json` + `forge_status.read_run_state`/`render_status` for an on-demand peek; the per-task stdout progress narrative (what you watch in the foreground TUI); `--timeout` as the hang backstop (a dead-man's switch — a stuck task is killed and escalated, so even a hang becomes a relayed halt). Supersedes the three prior Phase 6 session-awareness decisions (2026-07-14 session awareness, 2026-07-15 per-task notify, 2026-07-15 hook auto-wire) — those record the path taken; this records landing on the simpler model after Codex reality. Net: less code, no sandbox fight, and the halt-visibility need is met. Rejected: the monitor terminal (a separate out-of-sandbox process reading the run dir) — a real option for *push-while-away*, but the operator confirmed passive in-window visibility is all they need, so YAGNI.
**Where:** scripts/forge-run.py (fire_notify + --notify removed), scripts/forge_status.py (render_hook_block removed), hooks/hooks.json (UserPromptSubmit removed), hooks/user-prompt-submit + tests/test_prompt_hook.py + tests/test_forge_notify.py + tests/conftest.py (deleted), docs/forge/specs/2026-07-13-codex-exec-runner-design.md (§Session awareness + changelog), skills/planning/codex-execution.md, README.md

## 2026-07-15 — Phase 6 --notify gains a per-task `task-passed` event (modal per task)
**Why:** The original Phase 6 notify set fired only on terminal events (`escalated`, `contract-error`, `completed`), so a backgrounded multi-task run was silent through tasks 1..N-1 and pinged once at the end — the operator wanted a ping as *each* task completes (hands-free progress without tailing a log or sending a prompt to trigger the hook). On a multi-task run the final task fires both `task-passed` and `completed` (two modals); kept as-is because both signals are wanted (task done + run done). ## 2026-07-15 — Phase 6 UserPromptSubmit hook: shared hooks.json + in-hook harness gate (supersedes the ~/.codex manual install)
**Why:** The `session-start` hook proves the shared plugin `hooks/hooks.json` auto-wires on **both** harnesses (Codex reads it and sets `CLAUDE_PLUGIN_ROOT`) with zero manual step, and that a self-gating hook (emit only on a signal) is harmless where it shouldn't act. Rejected: manual `~/.codex` install (a step + stale-path risk); firing in both with no gate (user requires Codex-only); harness detection via env vars (`CLAUDE_PLUGIN_ROOT` is set on both, not discriminating).

## 2026-07-14 — Codex runner commits per passed task; clean tree required every invocation (phase 5)
**Why:** The runner never committed during a run, so the plan-level final review's `git diff HEAD` had no clean boundary — when the working tree started dirty (prior uncommitted phase work), every final review swept the entire accumulated cross-phase diff and judged unrelated code against this plan's spec, producing halt-grade findings the spec could never satisfy; re-runs compounded because fixes stayed uncommitted too. Root fix over a snapshot band-aid: the runner commits each passed task (`git add -A && git commit -m "forge: task N — <title>"`, empty stage skipped), so HEAD advances to a clean checkpoint after every task and `git diff base_commit` is exactly this plan's work. This restores the vertical-slice discipline the Codex runner had dropped and reaches parity with the Claude inline path (which already commits per task — emergent Claude Code behavior, not a forge convention, which is why the bug never surfaced there; the gap on Claude's *dispatch* path is tracked as roadmap Phase 8). Clean-tree precondition enforced at **every** invocation (first and resume), dirty → contract error exit 1: chosen over auto-reset-on-resume because a hard reset could silently destroy human fixes made between invocations — the runner never resets or stashes user work. Per-task review base becomes the prior commit (`_snapshot_worktree`/stash retired); final-review base is `base_commit`, persisted in `run.json` on first invocation and reused on resume so the diff spans the whole plan. Ledger annotation kept and rides in the task commit (git log is the parallel record). Env-gated skips (skip → exit 0 → counted green) addressed by a shared planning-skill authoring rule (assert infra present or make skip exit non-zero; skip ≠ pass) rather than fragile runner-side skip-marker detection — YAGNI, and harness-neutral so it also advances Phase 8. Rejected: snapshot-at-run-start (leaves the no-commit/no-slice/cumulative-diff problems intact), runtime skip detection, retiring the ledger annotation (bigger change than the bug needs).
**Where:** docs/forge/specs/2026-07-13-codex-exec-runner-design.md (Commit discipline + changelog), scripts/forge-run.py, skills/planning/SKILL.md (env-skip rule), skills/planning/codex-execution.md, README.md
