# Phase 2: Execution efficiency & scripts — design

Phase 2 of the 2026-07 upgrade cycle ([ideation](../ideas/2026-07-02-upgrade-cycle.md) §6–§7). Owns: plan-execution efficiency rules, reviewer integrity, the two pipeline scripts.

## 1. Planning skill — plan-time edits (`skills/planning/SKILL.md`)

- Task template gains optional line after **Files:** — `**Spec:** <section heading>, <section heading>` — the spec sections this task's worker needs, named by heading text (unique prefix acceptable; extractor matches case-insensitively). Written at plan time; omitted when the task needs no spec context. Drives mechanical brief extraction (§3).
- Decomposition guidance (File structure first / task ordering): minimize dependency chains — wall-clock = critical path, not task count; prefer decompositions sharing interfaces over sequence.
- **Tier** paragraph appends tier-down preference: interfaces and test cases fully enumerated → prefer the lower tier. (Experiment deferred: DEFERRALS 2026-07-02.)

## 2. Planning skill — Execution section rewrite

- Context-lifetime rule replaces the ≤3-task inline threshold: inline only when accumulated context is an asset — few tasks, later tasks build on seeing earlier work; otherwise dispatch, even for serial phases. Worker context is born, used, discarded; inline context compounds forever.
- File-referenced briefs: worker prompts carry the brief-file path + exact file paths, never pasted plan/spec content. Brief bounds reading: "read these N files and spec §X, nothing else." Briefs generated by `extract-brief.py`.
- All trivial-tier tasks batch into one `forge-light` dispatch, respecting `Depends on`.
- Thin orchestrator: workers return one-paragraph reports; diffs and review packets flow reviewer↔file via `review-packet.py`, never through orchestrating context.
- Orchestrator never pre-rates finding severity in review prompts.
- Rework guardrails (two sentences in skill text): review loops cap at 2 iterations, then escalate to user with findings; end-of-plan summary reports review-cycle counts per task.
- Multi-task plans: one broad final review on `forge-deep` after all tasks pass — integration issues per-task review can't see. Runs alongside the existing full-test-suite close-out.
- Close-out addition (one sentence): recurring "wouldn't have done it that way" review calls become written conventions — CLAUDE.md, or DECISIONS when architectural.
- Unchanged: routing table, combined-review + escalation flow, proportional review, deferral rule.

## 3. Scripts (`scripts/`, Python 3 stdlib only, no dependencies)

Inclusion test (constraint): a script must eliminate model *reading*, not typing. No memory-file CRUD tooling. A third script proposal → stop, re-justify.

### `scripts/extract-brief.py`

```
extract-brief.py <plan.md> <task-number> [--spec <spec.md>] [--out <dir>]
```

- Output: `<out>/task-<N>-brief.md`; prints path to stdout.
- Contents: plan header contracts (Goal, Global Constraints), full Task N block, spec sections named on the task's `**Spec:**` line (case-insensitive heading-prefix match against `--spec` headings; ambiguous prefix = error).
- Missing task number, missing `--spec` when task declares `**Spec:**`, or unmatched section heading → nonzero exit + message on stderr. Never emit a silently thin brief.

### `scripts/review-packet.py`

```
review-packet.py <plan.md> <task-number> --base <git-ref> [--out <dir>]
```

- Output: `<out>/task-<N>-review.md`; prints path to stdout.
- Contents: Task N block (interface, tests, acceptance) + `git diff <base>` output in a fenced `diff` block; fence length exceeds the longest backtick run in the diff body (min 3).
- Missing task number or failed git invocation → nonzero exit + message on stderr.

### Shared behavior

- `--out` defaults to system temp dir; orchestrator passes session scratchpad in practice. Output never committed.
- Task block parse anchor: `### Task <N>:` heading through the next `###`/`##` heading or EOF.

## 4. Agent edits (`agents/`)

- `forge-standard.md`, `forge-deep.md` review paragraphs add: review is read-only — never modify files; "can't verify from diff" is a valid verdict, report it as such; implementer rationales never suppress a finding.
- `forge-deep.md` review role widens: escalation reviewer **or** final integration reviewer (whole-plan diff against spec).
- `forge-light.md` unchanged — never reviews.
- Reviewer-facing rules live only in agent files; orchestrator-facing rule (no pre-rated severity) lives only in the planning skill. No relay of reviewer rules through orchestrator prompts.
